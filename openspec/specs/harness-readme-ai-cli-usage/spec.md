## 1. 需求规格（官方格式）

### 新增需求

#### 需求项：README 默认用户路径为 AI 工具 CLI + npx

系统必须在 README 中将目标项目用户入口统一为通过 AI 工具 CLI 触发 `npx @hunterzheng/harness`，npm install 和 npm run build 只能出现在开发者贡献章节。

##### 场景：目标项目用户首次使用
- **当** 目标项目用户首次阅读 README
- **预期** 系统（README）必须在前 50 行内展示 AI 工具 CLI 使用方式（Claude Code / Codex 内触发 npx），不得出现 `npm install` 作为用户安装步骤

##### 场景：README 安装章节
- **当** 用户阅读 README 的"安装"章节
- **预期** 该章节必须说明"目标项目用户无需手动安装，在 AI 工具 CLI 中输入 `npx @hunterzheng/harness` 即可自动完成初始化和安装"，不得包含 `npm install`、`npm run build` 等手动安装指令

##### 场景：README 使用章节
- **当** 用户阅读 README 的"使用"章节
- **预期** 必须说明"所有命令均在 AI 工具 CLI（如 Claude Code、Codex）内触发，底层由 `harness <command>` 调用"，不得把 `harness <command>` 展示为用户直接在终端执行的命令

#### 需求项：README 必须拆分用户路径和贡献者路径

系统必须在 README 中明确区分"目标项目用户使用方式"和"本仓库开发者贡献方式"，首屏和主要路径只讲 AI 工具 CLI + npx，npm 安装只保留为贡献者开发说明。

##### 场景：首屏内容拆分
- **当** 用户阅读 README 首屏
- **预期** 必须展示"## 使用方式（目标项目用户）"作为第一章节，内容指向 AI 工具 CLI + npx；"## 开发（本仓库贡献者）"作为独立章节，可包含 `npm install`、`npm run build` 等开发命令

##### 场景：贡献者路径不干扰用户路径
- **当** 用户阅读"贡献者路径"章节
- **预期** 必须明确标注"以下内容仅适用于 Harness 仓库本身的开发者，目标项目用户无需关心"，避免用户误认为需要克隆仓库

#### 需求项：AI 工具 CLI 使用说明完整

系统必须在 README 中提供 AI 工具 CLI 使用的完整说明，包括 Claude Code、Codex 等工具内的具体触发方式。

##### 场景：Claude Code 使用说明
- **当** 用户使用 Claude Code
- **预期** README 必须说明在 Claude Code 对话中输入 `npx @hunterzheng/harness` 即可触发 Harness 初始化向导，并展示向导流程概要

##### 场景：Codex 使用说明
- **当** 用户使用 Codex
- **预期** README 必须说明在 Codex 对话中输入 `npx @hunterzheng/harness` 即可触发 Harness 初始化向导

##### 场景：初始化后使用
- **当** 目标项目已完成 Harness 初始化
- **预期** README 必须说明后续所有命令（inspect/sync/develop/review/knowledge/status/doctor/config）均由 AI 工具 CLI 触发，用户只需在对话中表达意图，AI 工具会自动调用 Harness

#### 需求项：不对外暴露内部来源项目名

系统必须确保所有面向用户的命令说明不暴露 DocSync、GSD、kld-sdd、kld-review 等内部来源项目名作为日常命令。

##### 场景：命令列表不含内部来源名
- **当** 用户阅读 README 命令列表
- **预期** 命令列表只展示 `inspect`、`sync`、`develop`、`review`、`knowledge`、`status`、`doctor`、`config`，不得出现 `docsync`、`gsd`、`kld-sdd`、`kld-review` 等内部来源名称

##### 场景：CLI 帮助不含内部来源名
- **当** 用户执行 `npx @hunterzheng/harness --help`
- **预期** 帮助输出只展示上述 8 个命令，不得暴露内部来源项目名

#### 需求项：README 标注未完成状态的能力

系统必须在 README 中标注当前能力的完成状态，未完成或受限功能必须明确标记，不能让用户误以为已达到完整实现。

##### 场景：M1 能力状态标注
- **当** 用户阅读 README 能力列表
- **预期** inspect、sync、review 如为 M1 已完成，标记为"已可用"；develop 中 spec/design/tasks/check/apply/archive 多数阶段标记为"后续版本"；knowledge 标注"基础搜索可用，高级功能后续版本"；safety 标注"Hook 安装可用，完整编排后续版本"

##### 场景：受限功能说明
- **当** 能力存在受限边界
- **预期** README 必须在能力表格中备注限制条件，例如 review 的"当前为本地启发式扫描，多 agent 并行审查后续版本"
